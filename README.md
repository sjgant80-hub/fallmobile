# FallMobile

**The estate in your pocket.** A native Android + iOS app wrapping the AI-Native Solutions estate, adding what a browser can't do: **Bluetooth-LE mesh advertising and scanning** for phone-to-phone communication with no servers, no internet, no cell.

Part of the [AI-Native Solutions](https://ai-nativesolutions.com) estate. MIT licensed.

◊

## What FallMobile is

- A **Capacitor 7** wrapper around the estate portal (foldkit + FallCarrier + FallID + estate tools)
- A **BLE mesh transport** implemented as a native plugin, advertising + scanning a FallMobile service UUID
- **Bitchat-compatible** — decodes Bitchat wire format so users of both apps see each other on the mesh
- **Offline-first** — every core screen and tool works with no radios of any kind
- **Sovereign** — no telemetry, no analytics, no accounts, no cloud identity. Your DID lives in Preferences on your device
- **Full-screen** — no browser chrome, no address bar, safe-area aware, hardware back button wired

## Why it exists

The web platform will not let a page advertise a BLE service. Not on Chrome, not on Safari. Web Bluetooth is scan-only, and even scanning is behind a chooser dialog. **A phone-to-phone mesh needs the peripheral role**, which is only reachable from native code.

FallMobile takes the same estate you already run in a browser, drops it into a Capacitor shell, and exposes the peripheral role through the `@capacitor-community/bluetooth-le` plugin so **FallCarrier can register it as a transport**. Estate messages then route over Bluetooth transparently, alongside WebRTC, HTTP, or whatever else is available.

## Bitchat-compatible messaging

FallMobile advertises its own service UUID (`f0110000-1a1e-4a1e-8a1e-fa11ba11ba11`) **and** scans for the Bitchat service UUID (`f47b5e2d-4a9e-4c5a-9b3f-8e1d2c3a4b5c`). Incoming messages carry a **format byte**:

| Format | Meaning                             |
|:------:|:------------------------------------|
| `0x01` | FallMobile JSON envelope (FallCarrier native) |
| `0x02` | Bitchat wire-compat payload         |

The decoder in `src/fallmobile-transport.ts` translates Bitchat payloads into FallCarrier envelopes under topic `bitchat.msg`, so Bitchat users appear as ordinary mesh peers in the FallMobile UI.

## Build locally

You need: Node 20+, JDK 17, Android Studio (with Android SDK 34) for Android, macOS with Xcode 15 for iOS.

```bash
# 1. Install deps
npm install

# 2. Compile web assets (Vite bundles www/ + src/*.ts into www/dist/)
npm run build

# 3. Add native projects (first time only)
npx cap add android
npx cap add ios          # macOS only

# 4. Copy the built web bundle into the native projects
npx cap sync

# 5. Open in native IDE and build a signed APK / IPA
npx cap open android
# or
npx cap open ios
```

### Android permissions

Add these to `android/app/src/main/AndroidManifest.xml` inside `<manifest>`:

```xml
<uses-permission android:name="android.permission.BLUETOOTH" android:maxSdkVersion="30" />
<uses-permission android:name="android.permission.BLUETOOTH_ADMIN" android:maxSdkVersion="30" />
<uses-permission android:name="android.permission.ACCESS_FINE_LOCATION" android:maxSdkVersion="30" />
<uses-permission android:name="android.permission.BLUETOOTH_SCAN"       android:usesPermissionFlags="neverForLocation" tools:targetApi="s" />
<uses-permission android:name="android.permission.BLUETOOTH_ADVERTISE"  tools:targetApi="s" />
<uses-permission android:name="android.permission.BLUETOOTH_CONNECT"    tools:targetApi="s" />
<uses-feature android:name="android.hardware.bluetooth_le" android:required="true" />
```

### iOS permissions

Add to `ios/App/App/Info.plist`:

```xml
<key>NSBluetoothAlwaysUsageDescription</key>
<string>FallMobile uses Bluetooth to form a phone-to-phone mesh with nearby peers.</string>
<key>NSBluetoothPeripheralUsageDescription</key>
<string>FallMobile advertises presence over Bluetooth so nearby peers can discover you.</string>
<key>UIBackgroundModes</key>
<array>
  <string>bluetooth-central</string>
  <string>bluetooth-peripheral</string>
</array>
```

## Threat model + posture

**What FallMobile protects:**
- Message content between mesh peers (Ed25519-signed envelopes; content not encrypted at layer-1 — pair with an estate cipher for confidentiality)
- Identity across sessions (DID generated on-device, stored in secure Preferences, never phoned home)
- Estate work when offline (the whole app functions with airplane mode on)

**What FallMobile does NOT protect:**
- BLE presence — anyone within ~30m running a scanner can see that a FallMobile peer is nearby. Turn advertising off via the Mesh tab if that's a threat
- Traffic analysis at the radio layer — advertisement frequency, RSSI, and connection patterns are observable
- Compromised device — root/jailbreak on your own phone defeats every guarantee here

**Postures:**
- **Loud** — advertising + scanning both on. Best for discovering peers.
- **Listening** — scanning on, advertising off. You see them, they don't see you.
- **Silent** — both off. FallMobile behaves like a normal offline estate portal.

Toggle from the Mesh tab.
